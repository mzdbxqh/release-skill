# 03 -- README 质量

本文档定义 README 作为发布硬门的质量标准。`README.md`（英文）和 `README.zh-CN.md`（中文）均须满足以下要求。设计原则见 `00-target-state.md` P-7。

---

## 1. 首屏四问

README 首屏（不需要滚动即可看到的内容）必须让新用户回答以下四个问题：

| 编号 | 问题 | 验收标准 |
|---|---|---|
| Q-1 | 这是什么？ | 一句话定位，不含内部术语和架构术语 |
| Q-2 | 它适合谁、解决什么问题？ | 明确目标用户和问题域 |
| Q-3 | 默认会不会执行 push、GitHub Release 或 npm publish？ | 明确的安全承诺，说明外部写操作默认关闭 |
| Q-4 | 下一条最安全的可复制命令是什么？ | 可直接复制执行的命令，不依赖隐含前置条件 |

---

## 2. 正文结构

正文按用户任务组织，不按源码目录组织。推荐顺序：

1. 一句话定位和安全承诺（对应首屏四问）。
2. 支持的发布拓扑与不支持范围。
3. 五分钟安装和最小示例。
4. `release-help`、`assess`、`prepare`、`publish`、`reconcile` 的渐进式流程。
5. 配置最小样例。
6. 失败场景与恢复入口。
7. 安全、权限和供应链模型。
8. 完整 reference、贡献和维护者入口。

内部术语（如"发布单元""冻结计划""reconcile"）在首次出现时必须用用户语言解释。README 不得要求用户先理解父工程内部治理、状态机实现或 adapter 源码。

---

## 3. 三个黑盒角色

建立三个黑盒用户场景用于验收。黑盒执行者不得读取源码、设计文档或测试 oracle。若执行者需要猜测命令、默认权限或产物位置，README 验收失败。

### 3.1 首次使用者

- **输入**：只读 README，干净环境。
- **目标**：完成安装，发现 `release-help`，并对 fixture 执行只读 `assess`。
- **验收**：无需猜测任何命令或路径即可完成。

### 3.2 项目维护者

- **输入**：只读 README，一个父工程 + 单插件项目。
- **目标**：写出最小配置并完成 `prepare` dry-run。
- **验收**：配置格式、字段含义和 dry-run 命令均可从 README 推导。

### 3.3 安全审阅者

- **输入**：只读 README。
- **目标**：明确指出何时发生外部写操作、授权如何失效、部分发布如何恢复。
- **验收**：安全模型、授权失效条件和恢复路径在 README 中有清晰说明。

---

## 4. 可执行命令

### 4.1 提取与标注

README 中的 shell 命令通过以下方式标注以便自动提取：

```markdown
<!-- release-skill:exec fixture=<id> -->
```sh
release-skill assess --root ./my-project
```
```

### 4.2 执行规则

- 标注命令在隔离 fixture 副本中执行，不修改原始项目。
- 所有相对链接、标题锚点、文件路径和安装命令必须验证。
- 示例输出中的关键字段通过 snapshot 或结构断言验证，避免展示已失效界面。

### 4.3 命令新鲜度

- 发布前在打包后的公开快照上重复执行 README 测试，防止父工程内通过、公开包中缺文件。
- README 中公开 Skill 名称必须与实际插件清单和目录一致。

---

## 5. 中英文合同

中英文 README 的以下四项标记必须保持一致；允许文字表达不同，不允许行为合同漂移：

| 标记类别 | 说明 |
|---|---|
| `capability` | 支持的功能列表 |
| `command` | 可执行命令及参数 |
| `safety` | 安全承诺和写操作边界 |
| `version` | 版本信息 |

验证方法：比较机器可读的标记而非自然语言文本，报告每个语言中缺失的标记。

---

## 6. 打包后复测

发布前必须在打包后的公开快照上重复执行 README 测试：

1. 构建公开快照（见 `04-supply-chain.md`）。
2. 在快照上提取并执行所有标注命令。
3. 验证所有链接、锚点和文件路径。
4. 验证 Skill 名称一致性。
5. 验证中英文标记一致性。

若打包后测试失败，发布计划不得进入 PREPARED 状态。

---

## 7. 人工审阅门

自动化无法证明文字真正易懂，因此每个重要版本还需要一次独立 README 审阅。审阅至少检查以下内容：

| 审阅项 | 要求 |
|---|---|
| 开篇方式 | 以用户问题开篇，不以项目历史或架构术语开篇 |
| 最小路径 | 最小路径短且无隐含前置条件 |
| 危险命令说明 | 每个危险命令在执行前说明影响 |
| 失败提示 | 失败提示直接指向诊断或恢复步骤 |
| 可操作性 | 不存在"功能清单很多，但用户不知道第一步做什么"的情况 |

README 黑盒场景或人工审阅未通过时，发布计划不得进入 APPROVED 状态（见 `01-state-machine.md`）。
